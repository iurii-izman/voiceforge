# Roadmap: приоритет внедрения

Порядок — от «внедрять в первую очередь» к «лучше позже». Учтены зависимости, быстрые победы и стабильность ядра.


| Приоритет | Направление                                           | Рейтинг | Зачем в этом порядке                                                                         |
| --------- | ----------------------------------------------------- | ------- | -------------------------------------------------------------------------------------------- |
| **1**     | Шаблоны встреч в `analyze`                            | 93      | Схемы уже есть; один флаг `--template` и ветвление в router — быстрый выигрыш.               |
| **2**     | Обновление статусов action items по следующей встрече | 90      | Схемы готовы; замыкает цикл «действия → следующая встреча → обновление».                     |
| **3**     | Экспорт сессии (Markdown/PDF)                         | 82      | `history --id N` уже даёт данные; одна команда `export` — высокая польза при малых затратах. |
| **4**     | Выбор модели Ollama в конфиге                         | 70      | Одна настройка; развязывает руки под разное железо без хардкода.                             |
| **5**     | Документация «Первая встреча за 5 минут»              | 73      | Нет зависимости от кода; снижает порог входа до первых реальных сценариев.                   |
| **6**     | Отчёты по затратам (cost report)                      | 77      | Данные уже в БД (daily/period); добавить команду или расширить `history`.                    |
| **7**     | Явный язык для STT                                    | 81      | Передать hint в Whisper из конфига — лучше качество при малых изменениях.                    |
| **8**     | Расширенные e2e-тесты                                 | 76      | Стабилизирует всё последующее; логично после первых фич, чтобы тестировать их.               |
| **9**     | Стриминговый STT в CLI (listen)                       | 84      | Инфраструктура в демоне есть; вывести partial/final в терминал.                              |
| **10**    | Live summary во время listen                          | 88      | Опирается на стриминг + периодический analyze; после п.9.                                    |
| **11**    | Управление PII (вкл/выкл, только email)               | 68      | Гибкость без больших изменений; после того как базовый поток закреплён.                      |
| **12**    | Простой локальный Web UI                              | 74      | Альтернатива CLI; разумно после стабильного набора команд.                                   |
| **13**    | Десктопный UI (Tauri)                                 | 95      | Высокий impact, но большой объём; после твёрдого ядра и, при желании, export/reports.        |
| **14**    | Офлайн-пакет (Flatpak/AppImage)                       | 85      | Упаковка после стабилизации ядра и хотя бы части фич выше.                                   |
| **15**    | Smart trigger по умолчанию в daemon                   | 55      | Поведенческое изменение; лучше после накопления отзывов по ложным срабатываниям.             |
| **16**    | Бот (Telegram/Slack)                                  | 72      | Нишевый сценарий; после того как сам продукт и отчёты устоялись.                             |
| **17**    | Интеграция с календарём                               | 78      | Внешние зависимости и сложность; когда базовый flow уже привычен.                            |
| **18**    | RAG: новые форматы (ODT, RTF и т.д.)                  | 62      | Постепенное расширение индексации.                                                           |
| **19**    | Prompt caching для не-Claude                          | 58      | Исследование и точечные доработки; когда объём запросов растёт.                              |
| **20**    | Поддержка macOS / WSL2                                | 65      | Расширение платформ; последний шаг после стабильного Linux-ядра.                             |


---

Кратко по блокам:

- **1–4:** быстрые победы по коду (шаблоны, action-items, export, Ollama config).
- **5–7:** документация, отчёты, качество STT — без больших архитектурных решений.
- **8:** укрепление качества (e2e) перед следующими фичами.
- **9–10:** стриминг и live summary — связаны, идут подряд.
- **11–12:** настройки и альтернативный UI поверх текущего CLI.
- **13–14:** десктоп и упаковка — после твёрдого ядра.
- **15–20:** опциональные и платформенные темы — по мере необходимости.
