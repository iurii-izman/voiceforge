# Рефлексия по проекту и совместной работе (2026-03)

Мысли в целом по проекту VoiceForge, по нашей с тобой работе (пользователь + агент), что можно улучшить, слабые места и общий взгляд.

**Дата:** 2026-03-14. **Версия:** 1.0.0-beta.1.

---

## 1. Проект в целом

**Сильные стороны:**

- **Чёткая архитектура и контракты:** D-Bus, envelope, config, runbooks — всё зафиксировано. Легко понять, где что живёт и что можно менять без поломки границ.
- **Документация:** 15+ runbooks, DOCS-INDEX, copilot-program-map, audit. Агент может ориентироваться по docs и не «угадывать» политику.
- **Разделение KD/KC/KV:** Решения (KD), реализация (KC) и ручные gate (KV) разделены. Это уменьшает риск, что агент начнёт что-то, требующее твоего решения.
- **Качество и безопасность:** keyring, pip-audit, bandit, gitleaks, pre-commit. Релиз через скрипты (check_release_metadata, check_release_proof) — честная граница «что гарантировано».

**Слабые места:**

- **Daily Driver Score 35/100:** Технически многое готово (Engineering ~80), но продуктово — setup за 2 минуты, «запустил и забыл», один сценарий без трюков — ещё не достигнут. Разрыв именно в productization, а не в фичах.
- **Cognitive complexity и крупные функции:** main.py, daemon.py, preflight.py, copilot-overlay.js — большие блоки с высокой сложностью. Рефакторинг без изменения поведения трудоёмок и рискован; Sonar будет «ругаться», пока не разобьём на меньшие функции.
- **Зависимости и алерты:** Dependabot 3 уязвимости (1 high, 2 moderate). Решение — за тобой (обновить / принять риск / задокументировать).
- **Два KC заблокированы:** KC11 и KC13 зависят от KV1/KV5. Пока ты не сформулируешь решение, автопилот не может их трогать — по дизайну правильно, но «очередь» KC заканчивается на KC14.

## 2. Совместная работа (пользователь + агент)

**Что работает хорошо:**

- **Один блок за раз, чёткий конец:** «Довести до конца: код → тесты → docs → commit + push → next-iteration-focus» — предсказуемый цикл. Агент не разбрасывается по десяти задачам.
- **Промпты и контекст:** Явное указание «KV1 разрешён — брать KC11» снимает неоднозначность. Приложенные runbooks (agent-context, next-iteration-focus, copilot-program-map) дают достаточно контекста без перечисления всего кода.
- **Правила репозитория:** Conventional Commits, не расширять scope блока, UI/UX баги в отдельные issues — агент следует им и не «расползается» по scope.

**Что можно улучшить:**

- **Повтор одного и того же промпта:** Если несколько раз подряд отправлять «бери следующий блок на автопилот» без разрешения gate, агент будет каждый раз отвечать «нет разблокированного KC». Имеет смысл один раз явно выбрать: «KV1 разрешён — KC11» или «сделай Sonar/релиз/аудит» и т.п., чтобы не тратить ходы.
- **Объём одного запроса:** Запросы вроде «Sonar + релиз beta 1.0 + аудит + что нужно от пользователя + рефлексия» — большие. Разбивать на 2–3 сессии (например: 1) Sonar + релиз, 2) аудит + документы, 3) рефлексия) может дать более стабильный результат и меньше риска таймаутов.
- **Теги и релиз:** Пуш тега `v1.0.0-beta.1` и создание Release обычно делаются после твоего подтверждения, что артефакты и CHANGELOG тебя устраивают. Агент может подготовить всё (версия, CHANGELOG, runbook), но финальный `git push origin v1.0.0-beta.1` — по твоей команде или явному «можно пушить тег».

## 3. Слабые места (кратко)

| Область | Слабость | Возможное улучшение |
|---------|----------|----------------------|
| Productization | Daily Driver 35%, нет «запустил и забыл» | Отдельный фокус: setup wizard, one-shot, defaults, auto-start (без расширения KC scope). |
| Сложность кода | S3776 в main/daemon/preflight/overlay | Планомерный рефакторинг по одному модулю; вынести в #165 или отдельные issues. |
| Gate KC11/KC13 | Два блока стоят | Сформулировать решение по KV1 (wording) и KV5 (platform) — тогда автопилот снова сможет брать KC. |
| Документация | Residual drift | Периодически сверять DOCS-INDEX и runbooks с кодом (например, в next-iteration-focus «проверить drift» раз в N итераций). |
| Тесты | OOM при полном pytest, omit для части модулей | Targeted tests по изменённому; полный прогон — в CI или при явном запросе. |

## 4. Что можно улучшить в процессе

- **Чёткий «следующий шаг» в next-iteration-focus:** Всегда один из вариантов: (1) «KV1 разрешён → KC11», (2) «KV5 разрешён → KC13», (3) «Sonar/hardening/релиз/аудит» с коротким списком подзадач. Так следующий чат сразу знает, что делать.
- **Короткий «статус одной строкой» в начале next-iteration-focus:** Например: «Версия 1.0.0-beta.1. KC14 выполнен. Следующий executable KC: нет (KC11/KC13 за gate).» — чтобы не перечитывать весь файл.
- **Релиз:** Зафиксировать в runbook: «тег пушит человек после проверки артефактов» или «агент пушит тег по явной команде» — и придерживаться этого.

---

## 5. Итог

Проект в хорошем состоянии для беты: контракты, доки, Copilot-волна (KC2–KC14 кроме KC11/KC13), релиз 1.0.0-beta.1 подготовлен. Главные ограничители сейчас — не код, а **твои решения** (KV1, KV5, приоритеты, Dependabot) и **продуктовая доводка** (Daily Driver). Автопилот хорошо справляется с «взять один KC и довести до конца» и с подготовкой релиза/аудита; слабее — в больших разнородных запросах за один раз и в том, что не может за тебя разрешить gate. Чёткое разделение «что агент делает» и «что нужно от тебя» (как в [what-user-must-do.md](what-user-must-do.md)) уменьшает взаимное недопонимание и повтор одних и тех же ответов.
